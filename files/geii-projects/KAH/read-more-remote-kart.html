NEC PROTOCOL DESCRIPTION
The accumulator wasn‚Äôt supposed to be fixed on the board, so it didn‚Äôt occupy any surface area on the board.
If going into details, here full requirements for transmitter board:
  • ‚óè Dimensions: board must be equal to 120mm (¬±1mm) in width and 80mm (¬±1mm) in length. It has 3mm (¬±0.5mm) mounting holes located in each corner to secure it to the rest of the transmitter's mechanical components. The centers of these holes are positioned 5mm (¬±0.5mm) from the edges of the PCB.
  • ‚óè Logo: it also should contain visible name of Bordeaux GEII department and the name of the design team.
Energy module
  • ‚óè Must use a 2S lithium-polymer battery to operate. This battery has a capacity that ensures the transmitter's operating autonomy is at least 60 minutes.
  • ‚óè Board must include a switch to turn its electronic circuit on and off.
Information acquisition module
  • ‚óè Transmitter includes two human-machine interfaces that control the kart's engine power and front wheel direction, respectively. These two interfaces are rotary and/or linear potentiometers with a tolerance of -/+1/32 of the stage's supply voltage. They must be ergonomically positioned on the transmitter to facilitate kart control.
  • ‚óè Klaxon: the transmitter includes a push button that the user can press to indicate that they wish to sound the kart's horn.
Information processing module
The transmitter includes a digital processing core that allows:
  • ‚óè to acquire analog information from the human-machine interfaces,
  • ‚óè to digitize this analog information and transform it into digital information,
  • ‚óè to encode the push button's digital data in 1 bit and the team's digital value in 7 bits, in order to create the "address" byte of the NEC protocol,
  • ‚óè to encode the engine power digital data in 3 bits and the wheel direction digital data in 5 bits, in order to create the "data" byte of the NEC protocol,
  • ‚óè to construct the transmission frames in accordance with the NEC protocol by integrating the previously calculated "address" and "data" bytes.
Repetitively: transmitter transmits an NEC frame each time the data is new (i.e., different from the previous frame), without any delay other than the minimum inter-frame delay imposed by the NEC protocol (frame transmission period of 108 ms). If the data in consecutive NEC frames is identical, the NEC frame transmission period is then set to 333 ms. The tolerance for these two frame transmission periods is -/+10%.
Action module
Power: the transmitter transmits protocol frames using an infrared transmission component. The peak infrared transmission power ensures that the transmitter can transmit infrared signals decodable by the receiver at a minimum distance of 10 m.
To ensure this transmission distance, it is necessary to apply a peak current greater than 200 mA per LED to at least two 5 mm infrared LEDs. Since the emission of an infrared LED is relatively directional, the LEDs are carefully placed and oriented on the transmitter's PCB to optimize the transmission distance and angle.
Indicator: the kart transmitter includes a light indicator (50mcd -/+20%) informing the user that the transmitter is powered on.

Designing board
Synoptic architecture of the developed product
First step is to choose components for the project. We read datasheets of proposed components, choosing some of them like possible components for project. For example, for information acquisition, we chose two linear potentiometers PTA4543.
PTA4543
We needed to chose two potentiometers and had choice: take two linear potentiometers, two rotary or one of each. As you can see on architecture synoptic, we decided to stop on two same potentiometers, since it should be more comfortable to control direction and speed with them.
Functionality is next: for the power (speed) potentiometer, if it (lever pin 2) is in the low position (pin 1), there is no power in the motor (zero power), if the lever is in the high position (pin 3), there is maximum power in the motor. For the wheel steering (direction) potentiometer, if it (lever, pin 2) is in the middle, the wheels do not turn, if it is on the left (pin 1), the wheels turn left, and if it is on the right (pin 3), the wheels turn right.
Final potentiometer diagram
Value of this potentiometers is 10kohm. To stabilize the voltage, we need to add a capacitor. To choose the capacitor value, we used the cutoff frequency formula:
Our frequency is 10Hz since signals will be sent every 100 milliseconds. Using this information and values that we have, we found value of capacitor C=1.591 ŒºF. With this, we took capacitor of 1 ŒºF ¬±20%.
We also performed some simulations, viewing expected results, like when we have potentiometer in low position of 0%, we got 0V on output. Or if we had it in position of 50%, in the middle, then we got 2.5V on output.
We also chose simple push-button MCDTS6-3K.
Since this button would be connected to the MCU, ATMEGA328P, we needed to include it into our calculations for resistor.
Microcontroller has its own voltage Vil that equals to 0.3*Vcc and a leakage current equal to 1¬µŒ©. If the voltage exceeds Vil, the microcontroller will read a logic "1" from the input, so we have to make sure that the voltage produced by the leakage current in the resistor is not higher than this value.
The final horn assembly
Using formula that we get to know during our studies in university, we get a value of 1.5 ùëÄ‚Ñ¶. In order not to have interference with the human hand we took resistance of 100K‚Ñ¶. With this, if there is no pressure on the button, 0V will be on output, and there will be 5V if button is pressed.
For actions part we chose two TSAL6200 infrared LEDs to send signals to receiver. Green LED HLM-CM3A-Z10DD was chosen to indicate that board is turned on.
Final power stage schematic
In details, TSAL6200 met the requirements in size and its peak current was 200mA. In the same time, the maximum current delivered by the chosen MCU, is 40 mA (per I/O pin). With this value, we cannot actually light up two infrared LEDs.
To resolve this issue, it was proposed to use assembly of NPN transistor with resistors (that were placed in series with the LEDs in the collector section and one in the MCU output that goes to the base).
During detailed design stage we chose BC337 transistor. Chosen values for resistor you can see on schematic above. We also took a capacitor to store current. Aside of successful simulations, we recalculated current, that should have been 200mA minimum. Result was 250mA that satisfied requirements.
Green LED assembly
For Green LED, we decided to not connect it to the MCU, and do usual assembly with resistor in order to protect it and adjust the light intensity. Looking in the datasheet of LED, we got intensity value for 20mA and recalculated values to get current for 50mcd.
Unfortunately, we forgot to do inverse calculation for intensity, so we couldn‚Äôt see our error before verification stage. We passed requirements and so this part was not confirmed in the end.

Later we finished programming with next code sections:
Initiation of files and declaration of pins and functions
Before coding, we defined all the libraries we'll use in the project, all the pins to which the components are connected, and we also declared all the program functions.
Main code function void loop()
Main is an acquisition part. The AcquireNECFrame() function is part of the NEC library and checks for errors in the received NEC frame. This function retrieves the address and data contained in the NEC frame.
If the error is zero (no error in the NEC frame), we check whether the team number is correct. If so, we light up the blue LED, indicating that the received NEC frame is valid. We can then call the functions that will calculate the values to use to control the kart. These functions are located in the processing section (below), and to send commands to the kart, we use the action section functions (below).
And finally, action part. You can see example of programming of buzzer and one of motors below:
function code DriveServoMotor()
This DriveServo() function is dedicated to generating a PWM signal that we will send to a servo motor using the .write() function from the Servo library, which allows boards to control a variety of servo motors. We also map() the direction between the servo limits. The map() function remaps a number from one range to another.
Power motor wasn‚Äôt in fact servo-motor, but we programmed it like it, following similar logic.
function code PiloterBuzzer()
The PiloterBuzzer() function is dedicated to generating a 4kHz square wave signal. The input value is either 1 or 0. If the buzzer state value is 1, the signal is generated for 0.125 milliseconds.
To do this, we use the tone() function, which generates a square wave of the specified frequency on a pin.
If the received value is 0, we do nothing, so we don't generate the sound. To do this, we use the noTone() function, which stops the generation of a square wave triggered by the tone() function. It has no effect if no tone is generated.

Firstly, we tested the receiver LED. For this, we needed generator, to turn on boards: developed receiver and transmitter, and oscilloscope.

For this, we followed next procedure: at the start, powered channel 2 of the generator with 5V and channel 1 with 7.4V. Take the black and red wires, connect the black wire to ground and the red wire to Vcc. 5V is assigned to the receiver, and 7.4V to the transmitter.

Test procedure
Before calculations, we verified value of the resistor by turning board off and measuring value with the multimeter in ohmmeter mode.
To calculate intensity, we referred to the Ohm‚Äôs law, calculating current with values obtained during test.
